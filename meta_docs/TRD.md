# FreightDoc — Technical Requirements Document (TRD.md)

## 1. Technical Summary
FastAPI backend + React/Vite/Tailwind PWA frontend. Six-stage pipeline
combining deterministic logic (rule engine, PDF rendering, external API
calls) with three LLM reasoning stages (classification, generation,
cross-validation) on GPT-5.6 Luna at runtime. Built via Codex sessions
using GPT-5.6 Terra.

## 2. Functional Requirements
- FR1: Accept product description + country pair + quantity + declared
  value as input.
- FR2: Classify HS code with confidence score via GPT-5.6 Luna.
- FR3: Retrieve tariff data from USITC HTS API, UN Comtrade, and EU TARIC
  (destination-dependent).
- FR4: Determine required documents via a JSON-configured rule engine keyed
  by country pair + product category.
- FR5: Generate all required documents as structured JSON in a single
  GPT-5.6 Luna call.
- FR6: Cross-validate the generated package for HS consistency, value/
  quantity mismatches, missing fields, and destination-specific compliance
  gaps; output a compliance score and ready-to-ship boolean.
- FR7: Render each document as a styled PDF via ReportLab.
- FR8: Expose all of the above via a single orchestrating endpoint,
  `POST /api/full-pipeline`, plus individual step endpoints for testing.
- FR9: Serve the frontend as an installable PWA.

## 3. Non-Functional Requirements
- NFR1: Full pipeline completes in under 20 seconds for a typical request.
- NFR2: No user data persists beyond the request/response cycle in the
  MVP (explicitly stateless).
- NFR3: External API failures degrade gracefully via cached fallback data,
  never a hard failure visible to the user.
- NFR4: All AI calls request structured JSON output; no free-form prose
  parsing in the critical path.

## 4. DOCUMENTATION ARCHITECTURE (MANDATORY)

### Ownership model
| Doc category | Owner | Update trigger |
|---|---|---|
| `docs/*` (public) | Whoever makes the change | Any change touching architecture, API surface, schema, security, or scaling |
| `.private_docs/*` (private) | Whoever makes the change | Any non-trivial design or rationale decision |
| Stage 1 docs (this set of 10) | Planning phase only | Frozen after Stage 2 begins, except for corrections |

### Documentation generation requirements
1. `docs/*` and `.private_docs/*` are generated by the **coding agent**
   (Stage 2), never by the planning LLM. The planning LLM's job (this
   document and its 9 siblings) is to specify precisely what must exist
   and why — not to write the content itself.
2. Documentation generation is a **per-phase deliverable** in `plan.md` —
   no phase is considered complete without its corresponding doc updates.
3. Every external integration, every prompt, every schema table, every
   security control, and every scaling decision must have a traceable
   entry in the appropriate `docs/*` or `.private_docs/*` file. See
   `architecture.md` for the full mapping of decision-type to file.
4. `.private_docs/` must be created and added to `.gitignore` in the same
   commit that first populates it.

### Documentation validation requirement
Before any phase in `plan.md` is marked complete, the coding agent must
verify:
- The relevant `docs/*` files reflect the current state of the code.
- The relevant `.private_docs/*` rationale files exist and are non-generic.
- `CHANGELOG.md` has an entry for the phase.

## 5. Technology Constraints
- Must use GPT-5.6 (Luna for runtime, Terra for Codex build sessions) per
  hackathon rules.
- Must use Codex as the coding agent throughout the build, not only at the
  start.
- Deployment must remain within free tiers (Railway backend, Vercel
  frontend) for hackathon budget reasons.

## 6. Dependencies
- OpenAI Python SDK v1.x
- FastAPI, Uvicorn, httpx, Pydantic, ReportLab, python-dotenv
- React, Vite, Tailwind CSS, vite-plugin-pwa

## 7. Traceability
Every FR and NFR above must map to at least one entry in `docs/hld.md`
(architecture level) and `docs/lld.md` (implementation level) once Stage 2
is complete.
