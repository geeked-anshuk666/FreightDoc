# FreightDoc login, pipeline, and hero recovery plan

## Scope and diagnosis

- The affected UI is `frontend/src/components/AuthExperience.jsx` and `frontend/src/components/auth-experience.css`: desktop auth uses `100dvh` but its form can exceed the viewport; mobile has a deliberately taller document flow. Preserve Clerk routing and form functionality while removing only unintended page scrolling.
- The hero is `frontend/src/components/FullscreenCargoStory.jsx` plus `frontend/src/components/fullscreen-cargo-story.css`. Its current manual `scroll` listener independently mutates CSS variables across a `460vh` section; containers leave the camera composition and no single timeline owns framing, which explains the blank navy intervals and isolated oversized cargo.
- The stateless demo endpoint is `POST /api/full-pipeline` in `backend/app/routers.py`; its `PipelineService` and `GroqClient` are in `backend/app/services/pipeline.py` and `backend/app/services/groq_client.py`. A `502 AI_SERVICE_ERROR` currently hides the failed AI stage and the frontend renders the structured `detail` object as `[object Object]` through `frontend/src/hooks/useFreightPipeline.js` and its `ShipmentDesk` caller.

## Phased implementation

1. **Capture a baseline.** Reproduce `/sign-in`, `/sign-up`, `/`, and one `/api/full-pipeline` request; record viewport size, response status/body, `X-Request-ID`, backend logs, configured provider/model (never secrets), and the precise pipeline stage that fails.
2. **Normalize API errors.** Add a small frontend error parser in `useFreightPipeline.js` that extracts `detail.message`, `detail.code`, request ID, and retry guidance from FastAPI payloads. Ensure no object is interpolated into text and display a human-readable actionable error in `ShipmentDesk.jsx`/`ErrorPanel.jsx`.
3. **Diagnose and harden the AI boundary.** In `groq_client.py`, retain the original exception type safely in structured logs and tag each classify/generate/validate call with its stage, model, attempt, and request ID. In `pipeline.py`/`routers.py`, return a safe stage-specific 502 contract (`code`, `message`, `stage`, `request_id`, `retryable`) without leaking prompts, keys, or raw provider responses; keep the existing one malformed-output retry and do not silently fabricate compliance results.
4. **Make pipeline recovery usable.** Surface a retry action in the form only for retryable failures, preserve form/upload state, show the correlation ID for support, and distinguish configuration/provider outage from invalid shipment input (422) and rate limiting (429).
5. **Constrain auth to one viewport.** Refactor `auth-experience.css` to make the desktop auth shell `height: 100dvh; overflow: clip`, size the Clerk region with available-height flex/grid rules, and reduce nonessential decorative spacing. Keep content scroll-free at common desktop heights; use a deliberate compact mobile layout that may scroll only when the keyboard or Clerk verification/error content physically requires it.
6. **Keep Clerk reliable.** Do not alter `SignIn`/`SignUp` routing, OAuth triggering, redirects, or accessibility. Verify the auth panel has visible focus/error states and that its contained form never gets clipped; prefer CSS variables and supported Clerk appearance hooks over global internal selectors where possible.
7. **Replace, do not patch, the hero controller.** Rebuild `FullscreenCargoStory.jsx` around one GSAP `ScrollTrigger` master timeline (`scrub: true`) attached to a single `400–500vh` story section with one sticky `100dvh` viewport. Remove the custom scroll listener and all per-container event-driven progress calculations.
8. **Build one coherent scene graph.** Keep sky, distant port/cranes, wordmark, midground freight, foreground containers, vessel, copy, and transition overlay in one absolute full-viewport coordinate system. Start with a complete composed terminal; use only GPU-friendly `translate3d`, `rotate`, `scale`, and opacity transforms.
9. **Author cinematic beats on the single timeline.** Stage 0–20%: subtle camera/layer depth and complete hero. 20–55%: a suspended container descends while another crosses laterally; foreground/midground/background travel at clearly different rates. 55–78%: containers converge into an organized cargo-flow/vessel composition. 78–100%: copy fades gracefully, cargo moves outward/pulls away, arrival copy appears, and the next shipment section is revealed only after the pinned story completes. All progress must scrub backwards cleanly.
10. **Guarantee composition at every progress interval.** Define named keyframe poses for 0%, 25%, 50%, 75%, and 100% and test each at desktop dimensions. Keep at least two meaningful scene elements in frame at each pose; prohibit unintentional full-screen isolated cargo, empty navy screens, or content scaled below readable size.
11. **Implement responsive and reduced-motion variants.** Desktop receives the full timeline. Tablet lowers container count and distances while retaining the narrative. Mobile uses fewer cargo objects and a shorter pinned sequence; `prefers-reduced-motion` renders a stable finished cargo composition with a direct CTA and no pinned scrub sequence.
12. **Verify end-to-end.** Add/update backend tests for stage-specific AI error envelopes and retryable/non-retryable handling; add frontend tests or lightweight contract tests for error parsing, auth viewport overflow behavior, reduced motion, and hero timeline cleanup. Run backend tests, frontend production build, and manually inspect hero at 0/25/50/75/100% scroll plus sign-in/sign-up at desktop, tablet, and mobile sizes.

## Acceptance criteria

- `/sign-in` and `/sign-up` fit a normal desktop viewport without a page scrollbar, while Clerk fields, errors, OAuth, and keyboard accessibility remain usable.
- A failed full pipeline request never displays `[object Object]`; it identifies a safe next action and correlation ID.
- The hero has one full-screen pinned scene and one reversible scrubbed timeline, with no blank viewport interval or bounded animation frame.
- The form follows naturally after the cinematic sequence, and mobile/reduced-motion users receive a performant, readable alternative.
